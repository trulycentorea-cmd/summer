> PRD(`AI_발표_코치_PRD.md`)를 검토해서 **2일 안에 실제로 동작하는 결과물**을 만들기 위한 실행 계획입니다.
> 대상: 프로그래밍 초보자 / 기간: 2일 / 목표: "데모 가능한 완성품"

> **✅ 구현 완료 (2026-07-15).** P0 6개 + P1(레벨·배지·코치 캐릭터)까지 만들었습니다.
> 채점·코칭·눈맞춤 수학·오프라인 대비를 Chrome 자동 테스트 74개로 검증했습니다.
> **아직 안 한 것:** 실제 카메라·마이크·음성인식으로 직접 발표해보는 검증(장치가 필요해 자동화 불가) — Live Server로 열어 눈으로 확인하세요.
> **P2(캐릭터 꾸미기 상점·교사 대시보드)는 계획대로 만들지 않았습니다.**

## 1. 결론 먼저

PRD의 기능을 전부 만들면 2일로는 불가능합니다. 그래서 이렇게 자릅니다.

**만든다 (P0):** 발표 미션 체크 → 실제 카메라·마이크 발표 녹화 → 5가지 요소 별점 분석 → 코칭 문장 → 쿠키 보상 → 성장 그래프

**나중에 (P2):** 캐릭터 꾸미기 상점, 교사용 대시보드, 배지, 로그인, 서버

**가장 큰 결정 3가지**

| 결정 | 선택 | 이유 |
|------|------|------|
| 플랫폼 | **웹앱** (앱 아님) | 앱스토어·빌드·에뮬레이터 없이 Chrome에서 바로 실행. 초보자가 2일 안에 갈 수 있는 유일한 길 |
| 빌드 도구 | **없음** (순수 HTML/CSS/JS) | React·Vite·npm 설치에서 반나절 날립니다. `index.html` 더블클릭이면 끝 |
| AI | **API 안 씀** (브라우저 내장 기능 + 규칙 기반) | API 키·요금·네트워크 오류 리스크 제거. 음성/얼굴 분석은 브라우저가 이미 해줍니다 |

> **"AI를 안 쓰면 AI 발표 코치가 아니지 않나요?"**
> PRD 5장의 핵심 가치는 *"AI가 발표 내용을 평가하는 게 아니라 발표 습관을 코칭한다"* 입니다.
> 즉 필요한 건 **측정(음성 크기·속도·눈맞춤·자세) + 규칙 기반 코칭 문장**이고, 이건 LLM 없이 더 정확하고 빠릅니다.
> 실제로 얼굴 landmark 추론에는 머신러닝 모델(MediaPipe)이 돌아가니 "AI 없음"도 아닙니다.

## 2. 기술 스택

| 무엇을 | 무엇으로 | 난이도 |
|--------|----------|--------|
| 화면 | HTML + CSS (Flexbox) | 쉬움 |
| 로직 | 순수 JavaScript (ES Modules) | 보통 |
| 목소리 크기 | **Web Audio API** — `AnalyserNode`로 소리 크기 측정 | 보통 |
| 말하는 속도 | **Web Speech API** — 음성을 글자로 바꿔서 `글자수 ÷ 시간` | 쉬움 |
| 눈맞춤 + 자세 | **MediaPipe Face Landmarker** (CDN) — 얼굴 각도·흔들림 | 어려움 ⚠️ |
| 자신감 | 위 4개의 가중 평균 (계산식) | 쉬움 |
| 저장 | **localStorage** (브라우저 저장소, 서버 불필요) | 쉬움 |
| 그래프 | **Chart.js** (CDN 한 줄) | 쉬움 |

**실행 환경: Chrome 필수.** Web Speech API가 Safari/Firefox에서 안 됩니다. 발표 때도 Chrome으로 시연하세요.

**⚠️ 중요:** `index.html`을 더블클릭하면 카메라·마이크가 차단됩니다(`file://` 프로토콜). 반드시 로컬 서버로 띄우세요:
```powershell
# VS Code 확장 "Live Server" 설치 후 우클릭 → Open with Live Server  (가장 쉬움)
# 또는 Python이 있다면:
python -m http.server 8000    # → http://localhost:8000 접속
```

## 7. 데이터 구조 (localStorage)

키 하나에 JSON 전부 넣습니다. `"presentation-coach-data"`

```js
{
  cookies: 24,              // 보유 쿠키
  totalCookies: 38,         // 누적 (레벨 계산용)
  records: [                // 발표 기록 배열
    {
      date: "2026-07-15T10:30:00",
      duration: 45,          // 초
      scores: { confidence: 4, voice: 3, speed: 2, posture: 5, eyeContact: 4 },
      raw: { volumeDb: -22, syllablesPerMin: 340, eyeContactRatio: 0.72, swayStdDev: 12 },
      cookiesEarned: 4
    }
  ]
}
```
`raw`(측정 원본값)를 같이 저장하세요. 별점 기준을 바꿔도 과거 기록을 다시 계산할 수 있습니다.

## 8. 별점 계산 기준

**초등 저학년 기준. 실제로 발표해보고 숫자는 조정하세요.** (이게 제일 중요합니다)

| 요소 | 측정값 | ⭐5 | ⭐3 | ⭐1 |
|------|--------|-----|-----|-----|
| **목소리** | 평균 음량 (dB) | -20 이상 | -35 | -50 이하 |
| **속도** | 음절/분 | 200~280 | 150~330 | 범위 밖 |
| **눈맞춤** | 정면 응시 시간 비율 | 80%↑ | 50% | 20%↓ |
| **자세** | 머리 좌우 흔들림 (표준편차) | 작을수록 좋음 | — | — |
| **자신감** | `목소리×0.4 + 눈맞춤×0.4 + 속도×0.2` − (침묵 비율 페널티) | — | — | — |

**핵심: 속도는 "빠를수록 나쁨"이 아니라 "너무 빠르거나 너무 느리면 나쁨"** — 범위 중심 채점입니다. 초보자가 자주 놓치는 부분.

### 눈맞춤 보정 절차 (실제 학생과 쓰기 전에 꼭)

눈맞춤 기준값(`js/vision.js`의 `YAW_LIMIT`, `PITCH_MIN`)은 사람 얼굴 생김새와 카메라 높이에 따라 달라집니다. 실제 기기에서 한 번 맞춰야 정확해집니다.

1. Live Server로 앱을 열고 발표를 시작합니다.
2. **F12**(개발자도구) → **Console** 탭을 엽니다.
3. 카메라를 **정면으로** 바라보면 `[보정] yaw=... pitch=... → 정면 ⭕` 이 0.5초마다 찍힙니다.
4. 정면을 볼 때 대부분 `⭕`가 나오고, 고개를 옆으로 돌리거나 숙였을 때 `❌`가 나오면 기준이 맞는 겁니다.
5. 정면인데도 자꾸 `❌`가 나오면 → `YAW_LIMIT`(기본 0.10)을 조금 키우거나 `PITCH_MIN`(기본 0.13)을 조금 낮춥니다.
6. 옆을 봐도 `⭕`가 나오면(너무 후하면) → 반대로 값을 좁힙니다.

> 초등 저학년은 몸이 작아 카메라와 가까우므로, 태블릿을 실제 발표 높이(아이 눈높이보다 약간 아래)에 두고 보정하세요.

**코칭 문장 규칙 (PRD 6.3 준수)** — 평가하지 말고 **행동을 지시**하세요.
| 별점 | ❌ 이렇게 쓰지 마세요 | ✅ 이렇게 쓰세요 |
|------|---------------------|-----------------|
| ⭐1~2 | "목소리가 작아요" (평가) | "교실 맨 뒷자리 친구에게 들리게 말해볼까요?" (행동) |
| ⭐3 | "보통이에요" | "문장 끝마다 한 번 쉬어 보세요" |
| ⭐4~5 | "잘했어요" | "친구들을 잘 바라보며 발표했어요!" |
| 성장 시 | — | "지난 발표보다 목소리가 더 커졌어요!" ← **최우선 출력** |

**초등 저학년 대상이므로:** 별 1개여도 절대 부정적 표현 금지. 항상 잘한 것 1개를 먼저 말하고 개선점 1개만 제시하세요. 개선점 5개를 한 번에 쏟아내면 아이가 위축됩니다.
